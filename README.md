# SAIGE-Tractor tutorial

A walkthrough for running an admixed-cohort GWAS with **SAIGE-Tractor** end-to-end on a small simulated dataset. By the end you will have:

-   generated a simulated admixed cohort (200 individuals, 3 ancestries, 60 variants),
-   packed the phased genotypes + local-ancestry calls into SAIGE-Tractor's input format,
-   fit the Step 1 null model,
-   run the Step 2 association test, and
-   read off ancestry-specific and joint p-values from the output.

Everything below uses only files generated by [`simulate_example.py`](./simulate_example.py).

## 1. What SAIGE-Tractor is, briefly

SAIGE-Tractor performs a genome-wide association test (GWAS) **separately for each local ancestry** at every variant in an admixed cohort, while accounting for sample relatedness and case/control imbalance — both of which break older ancestry-aware methods like Tractor (OLS).

For every variant `v` and every ancestry `j` (e.g. AFR / EUR / AMR) it reports:

-   the per-ancestry effect size `BETA_anc{j}` and `p.value_anc{j}` (does this variant matter, *on AFR haplotypes*, on *EUR haplotypes*, …?),
-   a single ancestry-shared `p.value_ancALL` (the standard, ancestry-collapsed signal),
-   and a joint p-value `P_cct_admixed` that combines the two via a Cauchy combination — robust to whichever regime dominates.

You only need to run the test once per cohort to get all of these.

## 2. Installation

Docker is the **recommended** path — it pins the exact R / SAIGE / htslib / OpenBLAS versions and is the only one we test against. The other options exist for environments where Docker is not available (HPC clusters, native MacBook development).

### Option A — Docker (recommended)

``` bash
docker pull --platform linux/amd64 lhu1/saige-tractor:1.4.9-bundled
```

> **The image is multi-architecture (`linux/amd64` + `linux/arm64`), but please use the amd64 variant.** We observed numerical issues with SAIGE 1.4.9's PCG convergence and variance-ratio estimation when running on native arm64 (Apple Silicon). Until upstream SAIGE adds first-class arm64 support, we recommend forcing the amd64 variant on every command:
>
> -   **Apple Silicon Macs (M1/M2/M3/M4)** — pass `--platform linux/amd64` to every `docker pull` and `docker run`. Docker Desktop emulates amd64 via QEMU; performance is acceptable for tutorial-scale work, slower (but still usable) for biobank-scale jobs. For production runs prefer an x86_64 Linux server or cloud VM.
> -   **Intel Macs / Linux x86_64 servers / cloud VMs** — the amd64 variant is native; `--platform linux/amd64` is harmless but optional.
> -   **Linux arm64 (e.g., Graviton, Ampere)** — for now stick with amd64 emulation via `--platform linux/amd64` until the arm64-native numerics are fixed.

This image is **self-contained** — Docker is the only host-side prerequisite. It ships:

-   `step1_fitNULLGLMM.R`, `step2_SPAtests.R`, `step3_LDmat.R`, `createSparseGRM.R` (R wrapper scripts) on `$PATH` at `/usr/local/bin/`,
-   `flare_subset_to_tractor_hybrid` and `estimate_mac_threshold` (C++ utilities) in `/scripts/bin/`,
-   the SAIGE R package (v1.4.9) and all R/C++ dependencies (pixi-managed conda env: R 4.5, OpenBLAS, RcppArmadillo, …),
-   **`bgzip`, `tabix`, `bcftools` from htslib** — for compressing and indexing input VCFs (bundled in `1.4.9-bundled`; were missing from earlier images, which is why this tag exists).

Test the install:

``` bash
docker run --rm lhu1/saige-tractor:1.4.9-bundled \
    step1_fitNULLGLMM.R --help 2>&1 | head -3
# expected: usage and options for the Step 1 wrapper script
```

For the rest of this tutorial we will assume Docker. To run any command in the container against your local working directory, prefix it with:

``` bash

```

## 4. Input file overview

Five files. Take 30 seconds to look at each.

**4a. Phenotype TSV (`phenotype.tsv`)**

Tab-separated, one row per individual, with a header. `IID` must match the sample IDs in the VCF.

```         
IID     age  sex  PC1      PC2      PC3      Y
S0000   35   0    -0.1256  -0.6511   1.5985  -0.4489
S0001   47   0    -0.6532  -0.2485   1.3443  -1.0543
S0002   72   1    -0.9821   0.0123   0.4112   2.1390
...
```

For a **binary** trait, code cases as `1` and controls as `0` in the trait column.

**4b. Phased genotype VCF (`genotypes.phased.vcf`)**

Standard VCFv4.2 with **phased** genotypes (`0|1`, not `0/1`). Must be sorted, indexed, and bgzipped for production use (see below). One line per variant.

```         
##fileformat=VCFv4.2
##contig=<ID=chr22>
##FORMAT=<ID=GT,Number=1,Type=String,...>
#CHROM   POS       ID    REF  ALT  QUAL  FILTER  INFO  FORMAT  S0000  S0001  ...
chr22    16000000  v1    T    A    .     PASS    .     GT      0|0    0|0    ...
chr22    16005000  v2    G    C    .     PASS    .     GT      0|0    0|0    ...
```

**4c. FLARE local-ancestry VCF (`localanc.flare.vcf`)**

Same VCFv4.2 frame as the genotype VCF, but with two extra `FORMAT` fields, one integer ancestry label per haplotype. `AN1` is the ancestry of the first (`|`-left) haplotype, `AN2` is the second. Labels are integers `0..K-1` and are documented in the header lines.

```         
##ANCESTRY=<ID=0,Name=AFR>
##ANCESTRY=<ID=1,Name=EUR>
##ANCESTRY=<ID=2,Name=AMR>
##FORMAT=<ID=AN1,Number=1,Type=Integer,...>
##FORMAT=<ID=AN2,Number=1,Type=Integer,...>
#CHROM   POS       ID  REF  ALT  QUAL  FILTER  INFO  FORMAT       S0000      S0001      ...
chr22    16000000  v1  T    A    .     PASS    .     GT:AN1:AN2   0|0:2:2    0|0:2:1    ...
```

In the row above, sample `S0000`'s first haplotype is painted **AMR** (`AN1=2`) and second haplotype is also **AMR** (`AN2=2`); `S0001`'s second haplotype is painted **EUR** (`AN2=1`).

The sample IDs and variant positions in this file **must match** the phased genotype VCF exactly — same order, same coordinates. The FLARE tool produces this format natively. If you used a different local-ancestry caller (RFMix, MOSAIC, ...) you need to convert its output to this format first.

**4d. PLINK files (`example_grm.{bed,bim,fam}`)**

Used by Step 1 only, to build the genetic relationship matrix and pick markers for the variance-ratio calibration. Any small SNP set with reasonable allele-frequency coverage will do; you do *not* need to use the same variants as the GWAS dosage file.

**4e. Sample list (`samples.txt`)**

Plain text, one ID per line, no header.

## 5. Step 0 — pack the phased VCF + FLARE local ancestry

The `tractor_hybrid` packer collapses both VCFs into a small set of binary files that SAIGE-Tractor reads in a single streaming pass. This is much faster than re-parsing two VCFs for every variant at test time.

**5a. Index the VCFs**

The packer reads bgzipped + tabix-indexed VCFs. The bundled image now ships `bgzip` and `tabix` (htslib 1.10.2), so this runs entirely inside Docker:

``` bash
docker run --rm --platform linux/amd64 -v "$PWD":/work -w /work lhu1/saige-tractor:1.4.9-bundled bash -lc "
  bgzip -c example_inputs/genotypes.phased.vcf > example_inputs/genotypes.phased.vcf.gz
  bgzip -c example_inputs/localanc.flare.vcf  > example_inputs/localanc.flare.vcf.gz
  tabix -p vcf example_inputs/genotypes.phased.vcf.gz
  tabix -p vcf example_inputs/localanc.flare.vcf.gz
"
```

> Why `bgzip -c file > file.gz` and not `bgzip -kf file`? The bundled htslib is 1.10.2 (Ubuntu 20.04 vintage) which predates the `-k`/`--keep` flag. The `-c` redirect-to-stdout form keeps the original VCF on disk and works on every htslib version. Newer htslib in your own environment lets you use either.

**5b. Run the packer**

The packer takes five positional arguments — phased VCF, FLARE VCF, **number of ancestries K**, **MAC threshold**, and the output prefix:

``` bash
mkdir -p packed
docker run --rm --platform linux/amd64 -v "$PWD":/work -w /work lhu1/saige-tractor:1.4.9-bundled \
    /scripts/bin/flare_subset_to_tractor_hybrid \
        example_inputs/genotypes.phased.vcf.gz \
        example_inputs/localanc.flare.vcf.gz \
        3   \
        6   \
        packed/example
#       ↑   ↑
#       K   MAC threshold  (≈ N/32 — see note below)
```

| argument | meaning |
|------------------------------------|------------------------------------|
| `example_inputs/genotypes.phased.vcf.gz` | the phased genotype VCF |
| `example_inputs/localanc.flare.vcf.gz` | the FLARE local-ancestry VCF |
| `3` | **K**, the number of local ancestries (must match the labels in the FLARE header) |
| `6` | **MAC threshold** — variants with MAC ≤ threshold go to the sparse *rare-carrier* slab, the rest go to the dense *common* slab. See note below on how to pick it. |
| `packed/example` | output **prefix** — the packer writes `packed/example.meta`, `packed/example.common.geno.bin`, etc. |

> **Rule of thumb for the MAC threshold.** The dense common-variant slab stores roughly 1 byte per sample per ancestry block; the sparse rare-carrier slab stores \~8 bytes per non-zero carrier. The storage crossover (`8 · MAC ≈ N`) gives an optimum near **MAC ≈ N / 32**. For our N = 200 example that is `200 / 32 ≈ 6`. For UKBB (N ≈ 500k) it lands around MAC ≈ 15 000 (MAF ≈ 1.5 %), which also happens to be roughly where per-ancestry score tests stop being stable. For production data, run `/scripts/bin/estimate_mac_threshold` on the genotype VCF to refine this empirically — the rule of thumb gets you within a factor of 2.

Verify the packed output:

``` bash
ls packed/example.*
# packed/example.ancblock.bin    packed/example.common.geno.bin   packed/example.rare.variant.mks
# packed/example.ancblock.idx    packed/example.meta              packed/example.samples
# packed/example.ancblock.mks    packed/example.rare.carrier.bin
# packed/example.common.variant.mks
```

The single string `packed/example` is what you will pass to Step 2 as `--tractorHybridPrefix`.

## 6. Step 1 — fit the null GLMM

Step 1 fits the per-cohort phenotype model. **It does not use local ancestry** — local ancestry only enters at Step 2. Run it once per cohort + trait and re-use the output across all chromosomes.

``` bash
mkdir -p output

docker run --rm --platform linux/amd64 -v "$PWD":/work -w /work lhu1/saige-tractor:1.4.9-bundled \
    Rscript /scripts/step1_fitNULLGLMM.R \
        --plinkFile=example_inputs/example_grm \
        --phenoFile=example_inputs/phenotype.tsv \
        --phenoCol=Y \
        --traitType=quantitative \
        --invNormalize=FALSE \
        --covarColList=age,sex,PC1,PC2,PC3 \
        --qCovarColList=sex \
        --sampleIDColinphenoFile=IID \
        --outputPrefix=output/step1_example \
        --LOCO=FALSE \
        --IsOverwriteVarianceRatioFile=TRUE
```

Step 1 runs single-threaded — no need to set `--nThreads`. The fit will easily finish in seconds on this toy dataset.

| flag | meaning |
|------------------------------------|------------------------------------|
| `--plinkFile` | PLINK fileset prefix (no `.bed` suffix) |
| `--phenoFile` | phenotype TSV |
| `--phenoCol` | column name of the trait |
| `--traitType` | `quantitative` or `binary` |
| `--invNormalize` | rank-INT the trait before fitting (recommended for skewed quantitative traits; leave `FALSE` here because our `Y` is already roughly normal) |
| `--covarColList` | covariates to adjust for |
| `--qCovarColList` | subset of the above that are categorical |
| `--sampleIDColinphenoFile` | sample-ID column name in the phenotype file |
| `--outputPrefix` | output prefix; produces `.rda` and `.varianceRatio.txt` |
| `--LOCO=FALSE` | turn off leave-one-chromosome-out (only useful with whole-genome data; off for this 1-chromosome example) |

**Expected output**

```         
output/step1_example.rda                    <- fitted null model (binary R object; consumed by Step 2)
output/step1_example.varianceRatio.txt      <- variance ratio for Step 2 (one number)
```

A representative `varianceRatio.txt` and a representative console log are committed at [`example_outputs/step1_example.varianceRatio.txt`](./example_outputs/step1_example.varianceRatio.txt) and [`example_outputs/step1_example.log`](./example_outputs/step1_example.log) — useful for sanity-checking that your run finished in the same regime as ours. The `.rda` file is not committed because it is version-tied; regenerate it locally with the command above.

Step 1 should finish in well under a minute on this toy dataset. On real biobank data it is the slow part — typically the dominant cost of the whole pipeline.

> If you have related individuals you want to retain (you should), additionally pass `--useSparseGRMtoFitNULL=TRUE` plus a pre-computed sparse GRM via `--sparseGRMFile` and `--sparseGRMSampleIDFile`. See the SAIGE documentation (`createSparseGRM.R`) for how to build the GRM. We skip it here since this simulation has no related samples.

## 7. Step 2 — run the ancestry-aware association test

This is where local ancestry enters. The four admixed-specific flags are highlighted below.

``` bash
docker run --rm --platform linux/amd64 -v "$PWD":/work -w /work lhu1/saige-tractor:1.4.9-bundled \
    Rscript /scripts/step2_SPAtests.R \
        --tractorHybridPrefix=packed/example \
        --chrom=chr22 \
        --is_admixed=TRUE \
        --number_of_ancestry=3 \
        --pvalcutoff_of_haplotype=0.000005 \
        --GMMATmodelFile=output/step1_example.rda \
        --varianceRatioFile=output/step1_example.varianceRatio.txt \
        --SAIGEOutputFile=output/step2_results.tsv \
        --LOCO=FALSE \
        --minMAF=0 \
        --minMAC=1
```

The four flags that distinguish this from a standard SAIGE run:

| flag | meaning |
|------------------------------------|------------------------------------|
| `--tractorHybridPrefix` | the prefix from Step 0 (the packed input) |
| `--is_admixed=TRUE` | turn on the ancestry-aware machinery |
| `--number_of_ancestry=3` | K (must match what you packed in Step 0) |
| `--pvalcutoff_of_haplotype=0.000005` | trigger the local-ancestry-conditioned association test whenever any per-ancestry p-value falls below this. Set near genome-wide significance (≈ 5 × 10⁻⁶ to 5 × 10⁻⁸) for production runs. Setting it too liberally (e.g. 0.05) wastes compute on null variants. See section 8 for why the conditioned results are the ones you want to report. |

Everything else is standard SAIGE Step 2.

For this toy dataset, Step 2 should take a few seconds. The result is `output/step2_results.tsv`.

## 8. Reading the output

The output is one row per variant with a wide column structure. For K = 3 ancestries on a quantitative trait you get **71 columns**: 5 fixed + 4 × 15 per-ancestry-block (3 ancestries plus the `_ancALL` collapsed block) + 6 joint. The committed file [`example_outputs/step2_results.tsv`](./example_outputs/step2_results.tsv) is the **actual** Step 2 output from running this pipeline on the simulated data.

> **How the conditioning works (read this once).** At every variant, SAIGE-Tractor first runs a per-ancestry score test using the `j`-th ancestry-specific dosage as the test variable. It then runs a second, *preliminary* score test using the **count of haplotypes painted as ancestry `j`** as the test variable — this is the `Pvalue_haplo_anc{j}` column, and it asks "is the local-ancestry distribution itself associated with the phenotype at this site?" If `min_j Pvalue_haplo_anc{j} ≤ --pvalcutoff_of_haplotype`, the program then re-runs the per-ancestry score test **conditioning on the lead ancestry's haplotype-count vector** — these conditioned results are written to the `_c`-suffixed columns. If the min crosses the cutoff, the `_c` columns mirror their unconditional counterparts (the program just copies them through; `src/Main.cpp:5385–5392`). The cutoff therefore controls when the expensive marker-haplotype conditioning kicks in; setting it near genome-wide significance (5 × 10⁻⁶ to 5 × 10⁻⁸) keeps the compute cost down on real data.

> **Which p-value should you report?** **Always read off the `_c` columns.** When conditioning was triggered they are the local-ancestry-confounding-adjusted result (the value you want); when it was not triggered they equal the unconditional column, so reporting `_c` is correct either way. In particular `P_cct_admixed_c` is the recommended primary joint p-value and `BETA_c_anc{j}`, `p.value_c_anc{j}` are the recommended per-ancestry effect / p-value pair.

**Fixed columns (5)**

| column | meaning |
|-----------------------------------------------|------------------------|
| `CHR`, `POS`, `MarkerID`, `Allele1`, `Allele2` | variant identification. Note that `MarkerID` is written as `<id>_<ref>_<alt>` (e.g., `v31_G_A`) by the packer. |

**Per-ancestry columns — one 15-column block for each `j` in `1..K, ALL`**

For each ancestry block — `_anc1`, `_anc2`, `_anc3` (the K ancestries) and `_ancALL` (the collapsed / homogeneous dosage) — the writer emits this block, in order:

| column suffix | meaning |
|------------------------------------|------------------------------------|
| `AC_Allele2_anc{j}` | alt allele count on ancestry-`j` haplotypes only |
| `AF_Allele2_anc{j}` | alt allele frequency on ancestry-`j` haplotypes |
| `MissingRate_anc{j}` | per-ancestry missingness |
| `BETA_anc{j}` | per-ancestry effect size from the score test on ancestry-`j` dosage |
| `SE_anc{j}` | SE of the above |
| `Tstat_anc{j}` | per-ancestry score statistic |
| `var_anc{j}` | variance of the score statistic |
| `p.value_anc{j}` | per-ancestry p-value (SPA-corrected) |
| `BETA_c_anc{j}` | **report this**. Effect size after marker-haplotype conditioning when triggered, else = `BETA_anc{j}`. |
| `SE_c_anc{j}` | SE of the above |
| `Tstat_c_anc{j}` | conditioned score statistic |
| `var_c_anc{j}` | variance of the conditioned score |
| `p.value_c_anc{j}` | **report this**. Conditioned per-ancestry p-value, else = `p.value_anc{j}` |
| `N_haplo_anc{j}` | number of haplotypes painted as ancestry `j` at this variant (sanity check) |
| `Pvalue_haplo_anc{j}` | preliminary local-ancestry-haplotype-count score-test p-value. If `min_j Pvalue_haplo_anc{j} ≤ --pvalcutoff_of_haplotype`, the `_c` columns above were re-computed conditional on the lead ancestry's haplotype-count vector. This column is always populated; it is the *decision variable* for whether conditioning was triggered. |

For binary traits you additionally get `p.value.NA_anc{j}`, `Is.SPA_anc{j}`, `AF_case_anc{j}`, `AF_ctrl_anc{j}`, `N_case_anc{j}`, `N_ctrl_anc{j}` inside each block (and `_c` versions of the first two). The total column count for a binary trait with K = 3 is therefore higher than 71.

**Joint p-values (6 columns)**

| column | meaning |
|------------------------------------|------------------------------------|
| `P_het_admixed` | heterogeneity test — *ancestry-specific* effects |
| `P_hom_admixed` | homogeneous (collapsed) test — standard ancestry-agnostic p-value |
| `P_cct_admixed` | Cauchy combination of the two above |
| `P_het_admixed_c` | conditioned heterogeneity test (= `P_het_admixed` when not triggered) |
| `P_hom_admixed_c` | conditioned homogeneous test (= `P_hom_admixed` when not triggered) |
| `P_cct_admixed_c` | **conditioned Cauchy combination — the recommended primary p-value** |

### A worked example from the committed output

The actual `step2_results.tsv` produced by this pipeline on the simulated data flags the simulated causal variant `v31_G_A` (`chr22:16150000`):

```         
MarkerID    BETA_anc1   p.value_anc1   BETA_c_anc1   p.value_c_anc1   Pvalue_haplo_anc1
v31_G_A     +1.527      4.46e-13       +1.527        4.46e-13          1.51e-04

            BETA_anc2   p.value_anc2   Pvalue_haplo_anc2
            -0.166      0.684          0.288

            BETA_anc3   p.value_anc3   Pvalue_haplo_anc3
            -1.042      0.0455         0.0084

            BETA_ancALL  p.value_ancALL
            +0.903       2.46e-07

            P_het_admixed   P_hom_admixed   P_cct_admixed   P_cct_admixed_c
            2.91e-12        2.46e-07        5.82e-12        5.82e-12
```

How to read this:

-   `BETA_anc1` = **+1.53** with `p.value_anc1` ≈ **4 × 10⁻¹³** — a strong, highly significant effect on **AFR haplotypes**. This matches the simulation truth (`β_AFR_sim = 1.4`).
-   `BETA_anc2` and `BETA_anc3` are near zero / small — no real effect on EUR or AMR haplotypes (the small AMR estimate at `p = 0.045` is noise from only 6 AMR carriers; see `N_haplo_anc3 = 83`).
-   `p.value_ancALL ≈ 2 × 10⁻⁷` — the ancestry-collapsed test still picks the signal up, but the effect is **6 orders of magnitude weaker** than the AFR-specific one because the EUR and AMR haplotypes dilute it.
-   `Pvalue_haplo_anc1 = 1.51 × 10⁻⁴` — the preliminary haplotype-count test. Since `1.51 × 10⁻⁴ > 5 × 10⁻⁶`, the marker-haplotype conditioning **did not trigger**, so `BETA_c_anc1 = BETA_anc1` and `P_cct_admixed_c = P_cct_admixed`. Reporting the `_c` columns is therefore the same as reporting the unconditional ones — both give you `P_cct_admixed_c = 5.82 × 10⁻¹²` as the joint p-value.
-   On a real cohort with stronger LA confounding, you would expect to see variants where `Pvalue_haplo_anc{j}` does cross the cutoff — in those rows the `_c` columns will diverge from the unconditional ones and give you the LA-confounding-adjusted estimates.

The other 51 variants are non-causal: all their per-ancestry p-values are above 0.05, no `Pvalue_haplo_anc{j}` crosses `5 × 10⁻⁶`, and consequently all `_c` columns simply mirror the unconditional columns.

**Plain-English rule of thumb when scanning your own output:**

| if you see ... | what it means |
|------------------------------------|------------------------------------|
| `p.value_c_anc{j}` small for *one* ancestry only | ancestry-specific signal — report `P_het_admixed_c` and the per-ancestry `BETA_c`s |
| `p.value_c_anc{j}` small for *all* ancestries with consistent sign | shared signal across ancestries — report `P_hom_admixed_c` |
| `P_cct_admixed_c` is the smallest of the three joints | use it as the primary p-value; it is robust either way |
| `N_haplo_anc{j}` very small for some ancestry | that arm has too few haplotypes to estimate stably — ignore its `BETA_c_anc{j}` |
| `_c` columns identical to non-`_c` columns | `min_j Pvalue_haplo_anc{j} > --pvalcutoff_of_haplotype`, so marker-haplotype conditioning was not triggered. Reporting `_c` is still correct (it just reduces to the unconditional value). |

> **Want to see triggered `_c` columns on this toy dataset?** Re-run Step 2 with a more permissive cutoff, e.g. `--pvalcutoff_of_haplotype=0.001`, and `v31_G_A` will trigger conditioning (because `Pvalue_haplo_anc1 = 1.51 × 10⁻⁴ < 10⁻³`). In a production GWAS, leave the cutoff near genome-wide significance — the looser values are only useful pedagogically.

## 9. Common errors and how to fix them

| symptom | likely cause | fix |
|------------------------|------------------------|------------------------|
| `ERROR! bgenFile / vcfFile / ... does not exist` at Step 2 | wrong `--tractorHybridPrefix` (you passed a file name instead of the prefix without extension) | pass the prefix only, e.g. `packed/example`, not `packed/example.meta` |
| `chrom needs to be specified ... when using VCF or tractor_hybrid` | `--chrom` flag missing | pass the chromosome string exactly as it appears in the VCF — for our example: `--chrom=chr22`, not `--chrom=22` |
| Step 1 hangs or runs out of memory on related samples | full GRM is being built | pre-compute a sparse GRM and add `--useSparseGRMtoFitNULL=TRUE` |
| Step 2 errors out with `Sample IDs do not match` | phenotype-file IDs disagree with the packed VCF samples | recheck `samples.txt` in `packed/example.samples` matches the `IID` column in your phenotype file (and is a subset) |
| Many p-values are `NA` at low-MAC variants | per-ancestry MAC is below `--minMAC` | this is expected — rare variants split across K ancestries often fail the per-ancestry MAC cutoff; you can lower `--minMAC` or use the gene-based mode (in development) for rare-variant aggregation |
| Per-ancestry effect sizes look unstable for one ancestry | very few haplotypes painted as that ancestry at the variant | check `N_haplo_anc{j}` and `AC_Allele2_anc{j}`; if both are tiny, ignore that arm's effect estimate |
| `tractor_hybrid streaming step2 currently supports chromosome traversal only` | you passed `--idstoIncludeFile` or `--rangestoIncludeFile` | the packed format must be streamed; filter variants in the packing step instead, or post-filter the TSV output |
